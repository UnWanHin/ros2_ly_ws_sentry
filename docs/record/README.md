# 记录目录规范（docs/record）

本目录用于保存“已经发生的、会影响后续调试和接手”的重要变更记录。

目标：

- 让后续成员知道为什么改
- 让现场调试知道先看什么
- 让后续微调有统一入口

本目录不是：

- 纯想法草稿区
- 临时聊天纪要区
- 与代码无关的流水账区

## 适用范围

以下内容必须记录到本目录：

- 自瞄链路关键行为变化
- 开火判定/跟随判定变化
- topic / msg / 参数语义变化
- 上车配置默认值变化
- 明显影响实机表现的阈值改动
- 调试中确认过的回归点、风险点、临时约束

以下内容不要放这里：

- 纯计划草案
- 未落地的长期设想
- 无结论的随手猜测

这类内容应放到：

- `docs/plans/`

## 文件命名规则

统一使用：

- `主题_YYYY-MM-DD.md`

推荐示例：

- `autoaim_follow_fire_change_2026-03-18.md`
- `solver_geometry_fix_2026-03-20.md`
- `predictor_fire_gate_adjustment_2026-03-22.md`

要求：

- 名称能看出主题
- 日期必须写清楚
- 不使用 `new.md`、`temp.md`、`debug.md` 这类无信息文件名

## 每篇记录建议结构

至少包含以下小节：

1. `背景`
   - 现场表现是什么
   - 为什么需要改
2. `根因/判断`
   - 明确是已经确认的事实
   - 还是当前阶段的工程判断
3. `改动内容`
   - 改了哪些文件
   - 改动后的行为是什么
4. `关键参数/阈值`
   - 哪些可以从配置调
   - 哪些还在代码里
5. `微调建议`
   - 现场要快调时优先改什么
   - 不建议乱改什么
6. `验证方法`
   - 观察哪些 topic / 日志 / 现象

## 书写要求

- 尽量写“行为语义”，不要只写“改了某某文件”
- 尽量写“为什么这样改”，不要只写“改完能用”
- 如果某个值是临时保守值，要明确写出来
- 如果某个问题还没彻底解决，也要明确写出来
- 能区分“秒跟”和“稳打”的，不要混写

## 参数记录要求

涉及参数时，必须说明：

- 参数名
- 所在文件
- 当前默认值
- 调大/调小分别可能带来什么影响

如果某个关键阈值还在代码里，也必须注明：

- 文件路径
- 常量名或判断逻辑
- 为什么暂时不做成参数

## 与其他文档的关系

- 架构级说明放 `docs/architecture/`
- 包级说明放 `docs/modules/`
- 操作流程放 `docs/guides/`
- 变更落地记录放 `docs/record/`
- 长期方案放 `docs/plans/`

## 提交要求

凡是满足以下任一条件，代码提交时应同步更新本目录：

- 改了自瞄跟随/开火语义
- 改了默认配置
- 改了关键阈值
- 改了实机调参入口
- 修了一个会影响现场判断的坑

如果没有更新本目录，需要在提交说明里明确解释为什么不需要。
